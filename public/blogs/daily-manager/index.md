# Go 设计模式学习总结

> 学习主题：用 Go 的接口、组合、函数与并发原语理解创建型、结构型和行为型设计模式。

## 一、总体认识

设计模式是对常见软件设计问题的通用解法，不依赖具体语言。它的价值在于复用成熟经验、降低耦合、提高可维护性与扩展性；前提是识别真正的变化点，不能为了使用模式而增加不必要复杂度。

Go 不强调传统继承，模式通常通过 **小接口 + 结构体组合 + 函数注入** 实现。标准库中的 `http.NewRequest`、`bytes.NewReader`、`md5.New` 都体现了通过构造函数封装对象创建的思路。

## 二、创建型模式：管理对象如何创建

### 单例模式

目标是在进程内维护一个全局唯一资源，如配置、日志器或数据库连接池。实例应保持包内私有，对外仅暴露获取方法，并通常返回 `*T`：返回指针能让调用方共享同一对象，避免复制大型结构体；`nil` 也能表示尚未初始化。

```go
var (
    instance *Config
    once     sync.Once
)

func GetInstance() *Config {
    once.Do(func() { instance = &Config{} })
    return instance
}
```

`sync.Once` 保证并发首次访问时初始化函数只运行一次，并保证初始化结果对后续 goroutine 可见。若首次调用携带不同参数，先进入 `once.Do` 的参数生效；因此全局单例应在启动阶段统一初始化，不能让并发调用方决定配置。不同参数需要不同对象时，应使用普通构造函数或按 key 缓存的多例/享元方案。

### 简单工厂、工厂方法与抽象工厂

| 模式 | 核心做法 | 适用场景与代价 |
| --- | --- | --- |
| 简单工厂 | 一个函数用 `switch` 创建不同实现 | 产品少、规则简单；新增产品需修改分支，违反开闭原则 |
| 工厂方法 | 产品接口 + 工厂接口；每个具体工厂创建一种产品 | 产品类型持续扩展；类型数量增多 |
| 抽象工厂 | 一个工厂创建一组相互兼容的产品 | 需要产品族，如 Windows/Mac UI；新增产品类别需改所有工厂 |

工厂模式的本质是让业务依赖抽象产品接口，而不是具体类型。简单工厂是常用编码习惯；工厂方法适合“增加产品类型”；抽象工厂适合“成套切换相关实现”。

### 建造者模式

建造者模式将复杂对象的构造步骤拆开，适用于字段多、可选项多、创建顺序或校验规则复杂的对象。Go 中常见写法是链式 Builder 或函数选项（functional options），可避免大量构造参数导致的顺序混乱。

### 原型模式

原型模式通过复制已有对象创建新对象，适合初始化成本高、但对象间差异很小的情况。Go 中要区分浅拷贝与深拷贝：结构体直接赋值只复制字段；其中的 `map`、`slice`、指针仍可能共享底层数据，若要独立修改必须显式深拷贝。

## 三、结构型模式：组织对象与模块

### 代理模式

代理与真实对象实现同一接口，在访问真实对象前后加入控制逻辑，例如权限校验、缓存、延迟加载、远程调用或日志。代理关注“控制访问”；装饰器关注“叠加行为”。

### 桥接模式

桥接将两个独立变化的维度拆开并通过组合连接。例如 `Mac`、`Windows` 持有 `Printer` 接口，而不是定义 `MacHP`、`WindowsEpson` 等所有组合类型。它避免组合类爆炸，支持运行时替换实现，体现组合优于继承。

### 组合模式

组合模式用统一接口表示树形的“部分-整体”关系。文件与文件夹都实现 `Display`；文件夹持有 `[]FileSystemNode` 并递归调用子节点。客户端不用区分叶子和容器，适合目录、菜单、组织架构和评论树。

### 装饰器模式

装饰器与核心对象实现同一接口，并在内部包装该接口，在调用前后添加日志、监控、鉴权、缓存等能力。多个装饰器可嵌套，保持核心业务单一职责；Go 的 `io.Reader` 包装链是典型例子。

### 适配器与外观

- **适配器**：把第三方或旧系统的不兼容接口转换成系统期望的接口，例如将 `MakePayment` 转成统一的 `Pay`。重点是接口转换。
- **外观**：为库存、支付、物流等复杂子系统提供 `PlaceOrder` 这类简单入口。重点是简化调用和隐藏编排细节。

### 享元模式

享元通过共享可复用的内部状态来减少对象创建和内存开销，并把每次调用不同的外部状态独立传入。连接池是常见案例：复用连接而不是反复创建销毁。实现时需关注并发安全、资源归还、健康检查和关闭逻辑。

## 四、行为型模式：组织职责、流程与协作

### 责任链模式

请求沿多个处理器依次传递，每个处理器决定处理、拒绝或继续传递。适用于 HTTP 中间件、审批流、参数校验和过滤器链。它降低请求发送方与具体处理者的耦合，但链路过长时应保证顺序清晰、错误可追踪。

### 命令模式

命令模式将“要执行的动作”封装为对象或函数，使调用者与执行者解耦。它适合任务队列、菜单操作、重试、撤销/重做。Go 中函数本身常能充当轻量命令：`type Command func() error`。

### 解释器模式与抽象语法树

解释器模式为简单语言或规则定义语法结构，并解释表达式含义；表达式通常组织为抽象语法树（AST）。适用于筛选规则、简单查询语言、公式或权限表达式。复杂语言通常优先使用成熟解析器，避免自行维护庞大的语法树。

### 迭代器模式

迭代器以统一方式遍历集合，隐藏集合内部结构。Go 的 `for range` 已覆盖大多数场景；自定义迭代器适用于需要控制遍历顺序、分页、过滤或惰性读取的集合。

### 中介者模式

中介者将多个对象之间复杂的网状通信集中到一个协调者。例如聊天室由聊天室对象转发用户消息，表单协调器统一处理控件联动。它减少对象彼此直接依赖，但中介者不能演化成职责过重的“上帝对象”。

### 备忘录模式

备忘录在不暴露对象内部细节的前提下保存状态快照。原发器创建/恢复快照，管理者保存历史记录。适用于文本编辑器撤销、游戏存档、事务回滚。快照数据量较大时应控制数量或采用增量存储。

### 观察者模式

观察者建立一对多依赖：主题状态变化后通知所有已注册观察者。天气数据通知多个显示器是典型示例，适用于事件通知、消息订阅和领域事件。并发场景需保护观察者列表，并避免在持锁时执行耗时回调。

### 状态模式

状态模式将不同状态下的行为和转换封装到独立状态实现中，上下文把操作委托给当前状态。例如开关灯时 `OnState` 与 `OffState` 互相切换。它适合状态多、状态转换复杂的流程，可替代庞大的 `switch`。

### 策略模式

策略模式将可替换算法封装在同一接口下，由上下文在运行时选择。例如信用卡、支付宝、微信支付策略，或不同排序、压缩和路由算法。它解决“同一目标有多种实现”的问题；选择哪种策略的逻辑可放在调用方、配置或工厂中。

### 模板方法模式

模板方法固定算法骨架，把可变步骤交给具体实现。饮料制作中的烧水、倒入杯子是固定步骤，冲泡和调料是变化步骤；`WantCondiments` 可作为钩子控制可选流程。Go 无继承时，常通过接口和一个编排函数实现模板方法。

### 访问者模式

访问者把“操作”从稳定的对象结构中分离出来。文档中的文本、图片、表格实现 `Accept`，PDF/HTML 导出逻辑分别由不同 Visitor 完成。它便于频繁增加操作；但对象元素类型频繁增加时，需要修改 Visitor 接口及所有访问者实现。

## 五、核心辨析与收获

| 容易混淆的模式 | 区别 |
| --- | --- |
| 策略 vs 状态 | 策略由外部选择可替换算法；状态由对象内部状态驱动行为与转换 |
| 装饰器 vs 代理 | 装饰器增强功能；代理控制对真实对象的访问 |
| 适配器 vs 外观 | 适配器解决接口不兼容；外观简化多个子系统的使用 |
| 工厂方法 vs 抽象工厂 | 前者创建一种产品；后者创建一整组相关产品 |
| 观察者 vs 责任链 | 观察者是广播通知多个订阅者；责任链是请求按顺序传递处理 |

今天的重点是从“记住模式名称”转向“识别变化”：对象创建变化时考虑工厂/建造者；接口不兼容时用适配器；功能可叠加时用装饰器；流程有固定骨架时用模板方法；算法可替换时用策略；状态驱动行为时用状态；需要事件广播时用观察者。
